12  チームでの活用

🔵 中級者向け

12.1 チーム展開のロードマップ

Anthropic 社内の実践事例をもとにした、組織への展開フェーズです。

12.2 Phase 1: パイロット導入（1〜2ヶ月）

12.2.1 選定基準

  • 技術的な親和性が高いメンバーがいる部門
  • 業務が明確に定義されている（測定しやすい）
  • 失敗してもリスクが低い業務から始める

12.2.2 測定すべき指標

指標 測定方法
コード作成時間の短縮率 作業ログの比較
バグ発生率の変化 テスト結果・Issue数
コードレビュー時間 PR のマージまでの時間
メンバー満足度 アンケート

12.3 Phase 2: 水平展開（3〜6ヶ月）

12.3.1 共通の CLAUDE.md テンプレートを作る

チーム共通のルールを CLAUDE.md にまとめてリポジトリで管理します。

organization/
├── .claude/
│   ├── CLAUDE.md               # チーム共通ルール
│   ├── commands/
│   │   ├── commit.md           # コミット手順
│   │   ├── review.md           # レビュー手順
│   │   └── deploy.md           # デプロイ手順
│   └── agents/
│       ├── code-reviewer.md    # コードレビュー担当
│       └── security-audit.md   # セキュリティ監査

12.3.2 社内ナレッジベースの整備

## チーム共通 CLAUDE.md

# [チーム名] 開発ガイドライン

## テクノロジースタック
- バックエンド: Python 3.11 + FastAPI
- フロントエンド: React 18 + TypeScript
- インフラ: AWS ECS + RDS (PostgreSQL)
- CI/CD: GitHub Actions

## 開発フロー
1. Issue を確認してブランチを作成（`feature/issue-番号-概要`）
2. 実装 → テスト → レビュー依頼
3. `/review` でセルフレビューを実施してからPR作成
4. CI が通ったことを確認してマージ

## セキュリティルール（必須）
- API キーは絶対にハードコードしない（環境変数に）
- SQL は ORM を経由する（生SQLは禁止）
- 外部からの入力は必ずバリデーション

## 禁止事項
- `git push --force` to main
- 本番DBへの直接操作
- `.env` ファイルのコミット

12.4 チームAI-TDD（テスト駆動開発）

Anthropic が実践するAI強化型TDDです。

1. 要件の記述
   「ユーザー登録API：メール・パスワードを受け取り、バリデーション後にDBに保存する」
         ↓
2. Claude にエッジケースを網羅したテストを生成させる
   > この要件のテストコードを書いて。正常系・異常系・エッジケース全部含めて。
         ↓
3. テストが失敗することを確認（Red）
         ↓
4. Claude に最小限の実装を書かせる
   > テストを通す最小限のコードを書いて
         ↓
5. テストが通ることを確認（Green）
         ↓
6. Claude にリファクタリングさせる
   > SOLID原則に従ってリファクタリングして
         ↓
7. テストが引き続き通ることを確認（Refactor）

12.5 コードレビューへの活用

12.5.1 PR 作成前のセルフレビュー

> /review
（CLAUDE.md で定義したレビュー手順を自動実行）

12.5.2 レビュアーへの説明文生成

> このPRの変更内容を説明する文章を書いて。
  変更の背景、影響範囲、テスト方法を含めて。

12.6 新人オンボーディングへの活用

## 新人向け CLAUDE.md セクション

## 🚀 初日のセットアップ
1. このファイルを読む
2. `docs/ARCHITECTURE.md` を読む
3. Claude Code に「このプロジェクトの概要を説明して」と聞く

## 📚 学習リソース
- 技術スタックの学習: `docs/LEARNING_PATH.md`
- コードの読み方: Claude に「このファイルを説明して」と聞く
- 困ったとき: 先に Claude に相談、それでも分からなければメンターへ
