14  ハーネスエンジニアリング

🟣 上級者向け

14.1 ハーネスとは何か

「ハーネス（Harness）」とは、AIモデル本体以外のすべてを指す概念です。

具体的には：

  • 実行環境（ファイルシステム・シェルへのアクセス方法）
  • メモリアーキテクチャ（短期・長期記憶の管理）
  • ツール権限（何を許可し、何を禁止するか）
  • 検証機構（出力の安全チェック・自動テスト）
  • セッション管理（作業状態の引き継ぎ方法）

14.1.1 なぜハーネスが重要なのか

ImportantSWE-benchから判明した衝撃的な事実

研究（SWE-bench）によると：

  • 同じモデルでもハーネスを変えると 22ポイント のスコア変動が生じる
  • モデルを変えた場合の性能差は 約1ポイント に過ぎない

→ ハーネス設計は、モデル選択よりも重要

つまり、高価なモデルに切り替えるより、ハーネス（環境・設定・ツール構成）を適切に設計するほうが、AIの実際のパフォーマンスに大きく影響します。

14.2 GCTMSフレームワーク：エージェントの5つの構成要素

AIエージェントのアーキテクチャを車に例えた設計フレームワークです。

コンポーネント 車の比喩 役割
G — Guard（ガード） ブレーキ・ガードレール 危険操作を止める。実行前（入力）と実行後（出力）の二重構造で暴走を防ぐ
C — Context（コンテキスト） カーナビ 全情報を渡さず、「情報の在り処」と「現在地」だけを限定して提供する
T — Tool（ツール） ハンドル・ペダル ファイル操作・検索などの実行手段。入力・出力・エラー時のルールを明確に定義
M — Memory（メモリー） — 短期記憶（現セッションの会話履歴）＋ 長期記憶（過去の知識・ルール）
S — Supervision（スーパービジョン） ダッシュボード 成功率・失敗種類・トークン量・コストを監視し、改善サイクルを回す
Tipハーネス設計の効果

ハーネスを適切に設計することで、AIエージェントの成功率が約10倍向上した事例があります。

14.3 なぜハーネスなしでエージェントは失敗するか

Anthropicのエンジニアリングブログが記録した2つの典型的な失敗パターン：

パターン 内容
過剰スコープ 同時に多機能を試みてコンテキスト上限を超え、すべてが未完了になる
完了の誤認 再開時にエージェントが部分的な進捗を「完成済み」と誤解する

根本原因: LLMはセッションをまたいだメモリを持ちません。各コンテキストウィンドウはゼロからスタートするため、引き継ぎの仕組みがなければ状態が失われます。

14.4 ハーネスの主要コンポーネント

14.4.1 セッション間を橋渡しする必須ファイル

長期的な開発作業でエージェントを機能させるための基本ファイル構成：

ファイル 役割
claude-progress.txt 引き継ぎノート：完了した作業・次のタスク・既知の問題を記録
feature_list.json テストケースと合否状態の一覧（200件以上も管理可能）
init.sh 環境セットアップスクリプト。セッションごとに再現可能な環境を保証
Git履歴 安全な状態の時系列記録。コミットメッセージがエージェントの「記憶」になる

14.4.2 2エージェント構成パターン

大規模タスクでの推奨構成：

Initializer Agent（最初の1回）
  └── インフラ・進捗ファイル・feature list・初回コミットを作成

Coding Agent（2回目以降の各セッション）
  └── 進捗ファイルを読む → 1機能を選択 → 実装 → コミット → 記録更新

14.4.3 セッション開始の標準ルーティン

新しいセッションを始めるとき、エージェントに最初に実行させる手順：

1. git log で前回の作業を確認
2. 進捗ファイルを読み込んで現在地を把握
3. 基本機能が動作することを確認
4. 新しい作業を開始

14.5 ハーネスとCLAUDE.md

CLAUDE.mdはエージェントにとっての「プロジェクトの免許証」です。ハーネス設計における中核的な要素です。

14.5.1 CLAUDE.mdに記載すべき4項目

項目 内容 例
リポジトリの目的 1行で明確に このリポジトリはECサイトのバックエンドAPIです
禁止事項 絶対に変えない領域 本番DBの設定変更禁止、テストなしのマージ禁止
情報の在り処 詳細を書かずに場所だけ示す 設計書はDOCS、APIはopenapi.yaml
作業手順 AIが迷子にならないステップ 理解→小さく変更→テスト→差分説明

14.5.2 CLAUDE.mdのベストプラクティス

Warning長すぎるCLAUDE.mdは逆効果
  • 目標は最大50行
  • 150行を超えると「指示の初期バイアス」が発生し、後半の指示が機能しなくなる
  • 説明文ではなくポインタを書く：npm test を実行、ADRは /docs/adr/ にある
  • システム状態の詳細説明はしない → コードが信頼できる情報源

NG例（説明文）:

このシステムはNode.jsで構築されており、PostgreSQLデータベースを使用しています。
認証にはJWTを採用しており、リフレッシュトークンは7日間有効です...

OK例（ポインタ）:

## 情報の在り処
- 技術スタック: docs/ARCHITECTURE.md
- API仕様: openapi.yaml
- DB設計: docs/schema.md

## 禁止事項
- 本番DB（.env.production）の変更禁止
- テストが通るまでマージ禁止

14.6 フックアーキテクチャ：ハーネスの自動化機構

フックはハーネスの中核をなす自動化機構です（第9章参照）。ハーネス設計の観点から整理します。

14.6.1 フックの4種類と役割

フックタイプ タイミング ハーネスでの使い方
PreToolUse ツール実行前 安全ゲート・危険操作のブロック
PostToolUse ツール実行後 自動フォーマット・リント・テスト実行
Stop セッション終了前 テスト合格を完了条件として強制
PreCompact コンテキスト圧縮前 状態の保存

14.6.2 フィードバックループの速度階層

ハーネス設計では、フィードバックの速さを階層的に設計します：

PostToolUseフック（ミリ秒）
  ↓ ツールを実行するたびに即座に動作
pre-commitフック（秒）
  ↓ コミット前にチェック
CI/CD（分）
  ↓ プッシュ後にテスト全体を実行
人間レビュー（時間）
  ↓ プルリクエストで確認

→ フックを使うと「ミリ秒」レベルでのフィードバックが実現でき、AIがすぐに修正できる

14.6.3 ガードレールフックの実装例

危険な操作をブロックするPreToolUseフックの例：

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/scripts/safety-gate.sh \"$COMMAND\""
          }
        ]
      }
    ]
  }
}

safety-gate.sh の内容例：

#!/bin/bash
COMMAND="$1"

# sudo コマンドをブロック
if echo "$COMMAND" | grep -q "sudo "; then
  echo "ERROR: sudo は禁止されています" >&2
  exit 1
fi

# force push をブロック
if echo "$COMMAND" | grep -q "push --force\|push -f"; then
  echo "ERROR: force push は禁止されています" >&2
  exit 1
fi

# .env ファイルへの書き込みを防止
if echo "$COMMAND" | grep -q "\.env"; then
  echo "ERROR: .env ファイルの直接変更は禁止されています" >&2
  exit 1
fi

exit 0
Tipフックの終了コードの意味
  • 終了コード 0: 正常。Claudeは処理を続行する
  • 終了コード 1（または非ゼロ）: エラー。Claudeは処理をブロックし、エラーメッセージを受け取る

これにより、フックを使って「危険な操作をコードで止める」ことができます。

14.7 AIに特有のアンチパターンとハーネスによる対策

AIがコードを書く際に発生しやすい問題と、ハーネスで防ぐ方法：

アンチパターン 内容 ハーネスでの対策
型システム濫用 any 型の誤使用 リンタールールで強制検知
コード重複 既存コードを再利用せず新規生成 フック + コードレビューの自動化
ゴーストファイル 既存ファイルを修正せず類似ファイルを新規作成 ファイル追跡・PostToolUseフック
過剰コメント 90〜100%がAI生成コメント コメント密度チェック
セキュリティギャップ 36〜40%に悪用可能な脆弱性 PreToolUseゲートで事前ブロック

14.8 Plan → Work → Review サイクル

ハーネス駆動開発の典型的なワークフロー：

/harness-plan   → アイデアを実装計画と受け入れ基準に変換
      ↓
/harness-work   → 実装 + セルフレビューレポート
      ↓
/harness-review → 4視点分析（Security / Performance / Quality / Accessibility）
      ↓
/harness-release → CHANGELOG・タグ・リリースノートにパッケージ化

各ステップで人間の承認を挟むことで、AIが暴走するリスクを制御できます。

14.9 最小実行可能ハーネス（MVH）ロードマップ

すぐに始められる段階的な導入計画：

フェーズ 内容
Week 1 50行CLAUDE.md（ポインタのみ）+ pre-commitフック + PostToolUse自動フォーマット + 初回ADR
Weeks 2〜4 観察したエラーごとにテスト/ルール追加・plan→承認→実行ワークフロー・E2Eテストフレームワーク
Month 2〜3 ドメイン固有リンター・ADRとリンタールールの連携・古い散文ドキュメントの削除
Month 3+ マルチモデルチェーン・ガベージコレクションエージェント・有効性メトリクスの定量化
TipWeek 1から始める最小セット
  1. CLAUDE.md を作成（50行以内、ポインタのみ）
  2. pre-commitフックでlintを自動実行
  3. PostToolUseフックで自動フォーマット
  4. 進捗ファイル（PROGRESS.md）を作成してセッション間の引き継ぎを開始

14.10 ハーネス設計の3大原則

14.10.1 1. 薄く作る

AIモデルやツールの前提は半年で変わります。

❌ 巨大なフレームワークを構築する
✅ 最小限で始めて、柔軟に作り直せるようにする

14.10.2 2. 失敗を前提にする（多層設計）

1つのガードが突破されても、次の仕組みで止められるようにします。

PreToolUse（実行前ガード）
  ↓ 突破された場合
PostToolUse（実行後チェック）
  ↓ 突破された場合
CI/CDテスト（統合チェック）
  ↓ 突破された場合
人間レビュー（最終確認）

14.10.3 3. 人間をループに入れる

リスクに応じて3段階で区別します：

区分 内容 例
自動実行 リスクが低く、失敗しても復旧しやすい コードフォーマット、軽微なバグ修正
承認が必要 影響範囲が広い操作 新機能の追加、DBスキーマの変更
完全禁止 危険で取り返しのつかない操作 本番データの削除、force push

14.11 セキュリティの注意点

14.11.1 スキルとエージェントのリスク

外部から取得したスキルファイル（SKILL.md）を導入する場合は注意が必要です：

Warning外部スキルのセキュリティリスク
  • サプライチェーン攻撃: 悪意あるスキルがシステムを乗っ取る
  • 間接プロンプトインジェクション: 外部データを通じた指示の書き換え
  • シャドウAIデプロイ: 認可されていないエージェントの展開

対策: スキルファイルの内容を必ず人間が確認してから導入する

14.12 まとめ：ハーネスエンジニアリングのポイント

項目 要点
ハーネスとは モデル以外のすべて（環境・権限・検証・記憶）
重要性 モデル変更より22倍以上の影響力
GCTMSフレームワーク Guard / Context / Tool / Memory / Supervision
CLAUDE.md 50行以内、ポインタだけ書く
フックの使い方 速度階層に合わせて自動化レイヤーを設計
3大原則 薄く作る・失敗を前提にする・人間をループに入れる
